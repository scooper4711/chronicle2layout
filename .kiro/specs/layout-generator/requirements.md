# Requirements Document

## Introduction

The Layout Generator is a Python CLI tool (`python -m layout_generator`) that generates leaf layout JSON files for the Foundry VTT chronicle generator module. It reads a metadata properties file containing per-chronicle information (parent layout id, scenario id, description, defaultChronicleLocation), extracts checkboxes and item text from a chronicle PDF, and produces a layout JSON file with text-based choice parameters, presets, and content entries for item strikeouts and checkbox marks.

The tool is designed for batch processing across many chronicle PDFs. A metadata properties file is required because chronicles change their parent layout mid-season, so per-chronicle metadata cannot be inferred from directory structure alone. The tool resolves canvas regions from the parent layout (via `shared.layout_index`) to determine where on the PDF page to extract items and checkboxes.

## Glossary

- **Layout_Generator**: The Python CLI tool that generates leaf layout JSON files from chronicle PDFs and metadata.
- **Chronicle_PDF**: A single-page PDF file (or the last page of a multi-page PDF) containing a Pathfinder Society chronicle sheet with items and checkboxes.
- **Metadata_File**: A TOML file (default `chronicle_properties.toml`) containing an optional `layouts_dir` global property and an ordered array of `[[rules]]` entries. Each rule has a `pattern` (regex), `id`, `parent`, and optional `description` and `default_chronicle_location` templates with regex substitution support (`$1`, `$2`, etc.).
- **Parent_Layout**: An existing layout JSON file (e.g., a season base layout) that defines canvas regions, parameters, presets, and content inherited by the generated leaf layout.
- **Leaf_Layout**: The output layout JSON file generated by the Layout_Generator, containing only scenario-specific overrides (id, parent, description, defaultChronicleLocation, parameters, presets, and content for items and checkboxes).
- **Layout_Index**: A mapping of layout ids to file paths, built by scanning a layouts directory tree using `shared.layout_index.build_json_index`.
- **Canvas_Region**: A named rectangular area defined in a layout's `canvas` object, with percentage-based coordinates relative to a parent canvas.
- **Items_Canvas**: The Canvas_Region on the chronicle sheet where purchasable items are listed (typically named `items`).
- **Summary_Canvas**: The Canvas_Region on the chronicle sheet where adventure summary checkboxes appear (typically named `summary`).
- **Text_Line**: A dictionary containing extracted text and its bounding box as percentage coordinates (`text`, `top_left_pct`, `bottom_right_pct`) relative to the extraction region.
- **Item_Entry**: A segmented item with `text` (the item description including level and gold value), `y` (top y-coordinate as region percentage), and `y2` (bottom y-coordinate as region percentage).
- **Checkbox_Character**: A Unicode character representing a checkbox in PDF text: □ (U+25A1), ☐ (U+2610), ☑ (U+2611), or ☒ (U+2612).
- **Checkbox_Label**: A dictionary containing a detected checkbox's bounding box and its associated text label extracted from subsequent words in the PDF.
- **Strikeout_Preset**: A preset defining the horizontal line coordinates used to visually strike out an item on the chronicle sheet.
- **Checkbox_Preset**: A preset defining the position coordinates used to render a checkbox mark on the chronicle sheet.
- **Safe_Label**: A sanitized string derived from item text or checkbox label text, suitable for use as a preset name suffix (spaces replaced with underscores, punctuation stripped, truncated to 50 characters).

## Requirements

### Requirement 1: CLI Entry Point

**User Story:** As a layout author, I want to run the layout generator from the command line with a PDF file or directory, so that I can generate leaf layout JSON files for one or many chronicles.

#### Acceptance Criteria

1. THE Layout_Generator SHALL be invocable as `python -m layout_generator`.
2. THE Layout_Generator SHALL accept a required positional `pdf_path` argument that is either a path to a single Chronicle_PDF file or a directory containing Chronicle_PDFs.
3. WHEN `pdf_path` is a directory, THE Layout_Generator SHALL recursively process all PDF files in all subdirectories.
4. THE Layout_Generator SHALL accept an optional `--metadata-file` argument specifying the path to the Metadata_File, defaulting to `chronicle_properties.toml` in the current working directory.
5. THE Layout_Generator SHALL accept an optional `--layouts-dir` argument specifying the root directory containing Parent_Layout files for canvas resolution. WHEN not provided on the command line, THE Layout_Generator SHALL read the layouts directory from the `layouts_dir` property in the Metadata_File. IF neither is provided, THE Layout_Generator SHALL print a descriptive error to stderr and exit with a non-zero exit code.
6. THE Layout_Generator SHALL accept an optional `--output-dir` argument specifying the directory where generated Leaf_Layout files are written, defaulting to the resolved `--layouts-dir` value.
7. THE Layout_Generator SHALL accept an optional `--item-canvas` argument specifying the canvas name for item extraction, defaulting to `items`.
8. THE Layout_Generator SHALL accept an optional `--checkbox-canvas` argument specifying the canvas name for checkbox detection, defaulting to `summary`.
9. WHEN invoked with valid arguments, THE Layout_Generator SHALL process all matching entries and exit with code 0 on success.

### Requirement 2: Metadata File Format

**User Story:** As a layout author, I want to define pattern-based rules in the metadata file that match chronicle PDFs by regex, so that I don't have to list every single chronicle file individually.

#### Acceptance Criteria

1. THE Metadata_File SHALL be a TOML file containing an optional `layouts_dir` global property and an ordered array of rule entries under a `[[rules]]` table array.
2. EACH rule entry SHALL contain a `pattern` field: a regular expression matched against the Chronicle_PDF's path relative to the `pdf_path` directory (or the filename when `pdf_path` is a single file).
3. EACH rule entry SHALL contain a `parent` field specifying the parent layout id for matched chronicles. The `parent` field MAY contain regex substitution references (`$1`, `$2`, etc.) that are replaced with capture groups from the `pattern` match.
4. EACH rule entry SHALL contain an `id` field specifying a template for the generated layout id. The `id` field MAY contain regex substitution references.
5. EACH rule entry MAY contain a `description` field specifying a template for the layout description. The `description` field MAY contain regex substitution references.
6. EACH rule entry MAY contain a `default_chronicle_location` field specifying a template for the defaultChronicleLocation. The `default_chronicle_location` field MAY contain regex substitution references.
7. WHEN processing a Chronicle_PDF, THE Layout_Generator SHALL test the rules in order and use the first rule whose `pattern` matches the PDF's relative path.
8. IF no rule matches a Chronicle_PDF, THEN THE Layout_Generator SHALL print a warning to stderr and skip that file.
9. IF the Metadata_File does not exist at the resolved path, THEN THE Layout_Generator SHALL print a descriptive error to stderr and exit with a non-zero exit code.

### Requirement 3: Canvas Region Resolution

**User Story:** As a layout author, I want the tool to resolve canvas regions from the parent layout, so that it knows where on the PDF page to extract items and detect checkboxes.

#### Acceptance Criteria

1. WHEN processing a Metadata_File entry, THE Layout_Generator SHALL build a Layout_Index from the layouts directory using `shared.layout_index.build_json_index`.
2. THE Layout_Generator SHALL resolve the Items_Canvas coordinates by walking the Parent_Layout inheritance chain using `shared.layout_index.collect_inheritance_chain` and converting canvas percentages to absolute page percentages.
3. THE Layout_Generator SHALL resolve the Summary_Canvas coordinates using the same inheritance chain resolution.
4. IF the parent layout id from the Metadata_File entry is not found in the Layout_Index, THEN THE Layout_Generator SHALL print a descriptive error to stderr and skip that entry.
5. IF the Items_Canvas name is not found in the resolved layout canvases, THEN THE Layout_Generator SHALL print a warning to stderr and proceed without item extraction.
6. IF the Summary_Canvas name is not found in the resolved layout canvases, THEN THE Layout_Generator SHALL print a warning to stderr and proceed without checkbox detection.

### Requirement 4: PDF Text Extraction

**User Story:** As a layout author, I want the tool to extract text lines from the items region of the chronicle PDF, so that item descriptions can be segmented and included in the layout.

#### Acceptance Criteria

1. WHEN a Chronicle_PDF is provided, THE Layout_Generator SHALL open the last page of the PDF using PyMuPDF.
2. THE Layout_Generator SHALL extract words from the PDF page using PyMuPDF's text extraction.
3. THE Layout_Generator SHALL filter extracted words to only those within the resolved Items_Canvas region.
4. THE Layout_Generator SHALL group words into Text_Lines by quantizing their y-coordinates, treating words within 2.0 PDF points of each other vertically as belonging to the same line.
5. THE Layout_Generator SHALL convert each Text_Line's bounding box to percentage coordinates relative to the Items_Canvas region.
6. THE Layout_Generator SHALL sort Text_Lines by vertical position (top to bottom).
7. THE Layout_Generator SHALL skip lines whose text (lowercased, stripped) matches a bare "items" header.
8. IF the Chronicle_PDF has zero pages, THEN THE Layout_Generator SHALL return an empty list of Text_Lines.

### Requirement 5: Item Segmentation

**User Story:** As a layout author, I want extracted text lines segmented into individual items using parenthesis-based heuristics, so that each item gets its own strikeout line in the layout.

#### Acceptance Criteria

1. THE Layout_Generator SHALL segment Text_Lines into Item_Entries by streaming tokens across lines and tracking parenthesis depth.
2. WHEN two complete parenthesis groups have been closed and parentheses are balanced, THE Layout_Generator SHALL finalize the current Item_Entry and start a new one with any remaining tokens.
3. WHEN end-of-line is reached with balanced parentheses, THE Layout_Generator SHALL finalize the current Item_Entry.
4. WHEN end-of-line is reached with unbalanced parentheses, THE Layout_Generator SHALL continue accumulating tokens from the next line.
5. THE Layout_Generator SHALL clean item text by removing trailing uppercase-U artifacts after lowercase letters, removing hair-space characters (U+200A), and stripping leading/trailing whitespace.
6. THE Layout_Generator SHALL record each Item_Entry's vertical extent (y and y2) as percentages of the Items_Canvas region.
7. THE Layout_Generator SHALL skip lines whose cleaned text is empty.

### Requirement 6: Checkbox Detection

**User Story:** As a layout author, I want the tool to detect checkbox characters in the PDF text layer, so that checkbox positions can be included in the layout.

#### Acceptance Criteria

1. THE Layout_Generator SHALL scan the last page of the Chronicle_PDF for Checkbox_Characters (□, ☐, ☑, ☒) in the PDF text layer.
2. THE Layout_Generator SHALL filter detected checkboxes to only those within the resolved Summary_Canvas region.
3. THE Layout_Generator SHALL compute each checkbox's bounding box as percentage coordinates relative to the Summary_Canvas region.
4. IF no Summary_Canvas region is provided, THEN THE Layout_Generator SHALL compute checkbox positions relative to the full page.
5. IF the Chronicle_PDF has zero pages, THEN THE Layout_Generator SHALL return an empty list of checkboxes.

### Requirement 7: Checkbox Label Extraction

**User Story:** As a layout author, I want text labels extracted for each detected checkbox, so that the layout uses descriptive text choices instead of numeric indices.

#### Acceptance Criteria

1. WHEN checkboxes are detected and a Summary_Canvas region is provided, THE Layout_Generator SHALL extract text labels for each checkbox.
2. THE Layout_Generator SHALL collect label words by scanning subsequent words in the PDF text after each Checkbox_Character until a delimiter is reached.
3. THE Layout_Generator SHALL treat the following as label delimiters: another Checkbox_Character, the word "or" (case-insensitive), or trailing punctuation (comma or period).
4. WHEN a Checkbox_Character word contains additional text after the checkbox symbol (e.g., "□Label"), THE Layout_Generator SHALL include that text as the first label word.
5. THE Layout_Generator SHALL strip trailing punctuation (comma or period) from the assembled label, except for ellipsis ("...") or decimal numbers.
6. THE Layout_Generator SHALL produce a list of Checkbox_Labels, each containing the checkbox bounding box and the extracted label string.

### Requirement 8: Layout JSON Generation

**User Story:** As a layout author, I want the tool to assemble extracted items and checkboxes into a leaf layout JSON structure, so that the output conforms to LAYOUT_FORMAT.md and works with the chronicle generator module.

#### Acceptance Criteria

1. THE Layout_Generator SHALL produce a Leaf_Layout dictionary containing metadata fields: `id`, `parent`, `description`, and `defaultChronicleLocation` (each included only when the corresponding value is provided in the Metadata_File entry).
2. WHEN Item_Entries are extracted, THE Layout_Generator SHALL add an `Items` parameter group containing a `strikeout_item_lines` choice parameter with item text strings as choices.
3. WHEN Item_Entries are extracted, THE Layout_Generator SHALL add a base `strikeout_item` preset with the Items_Canvas name, color "black", and fixed x/x2 coordinates for strikeout lines.
4. WHEN Item_Entries are extracted, THE Layout_Generator SHALL add an `item.line.<Safe_Label>` preset for each item with the item's y and y2 coordinates (rounded to one decimal place).
5. WHEN Item_Entries are extracted, THE Layout_Generator SHALL add a choice content entry mapping each item text to a strikeout element referencing the base and item-specific presets.
6. WHEN Checkbox_Labels are extracted, THE Layout_Generator SHALL add a `Checkboxes` parameter group containing a `summary_checkbox` choice parameter with checkbox label strings as choices.
7. WHEN Checkbox_Labels are extracted, THE Layout_Generator SHALL add a base `checkbox` preset with the Summary_Canvas name, color "black", and size 1.
8. WHEN Checkbox_Labels are extracted, THE Layout_Generator SHALL add a `checkbox.<Safe_Label>` preset for each checkbox with its x, y, x2, y2 coordinates.
9. WHEN Checkbox_Labels are extracted, THE Layout_Generator SHALL add a choice content entry mapping each label to a checkbox element referencing the base and checkbox-specific presets.
10. WHEN both items and checkboxes are present, THE Layout_Generator SHALL order the output sections as: item parameters, item base preset, item content entry, checkbox parameters, checkbox presets, checkbox content entry, then item line presets.
11. IF no Item_Entries and no Checkbox_Labels are extracted, THEN THE Layout_Generator SHALL produce a Leaf_Layout containing only the metadata fields.
12. THE Layout_Generator SHALL skip Checkbox_Labels with empty label strings when building the checkbox parameter choices and content entries.

### Requirement 9: Output File Writing

**User Story:** As a layout author, I want the generated layout written to a JSON file in the output directory, so that I can use it with the chronicle generator module.

#### Acceptance Criteria

1. THE Layout_Generator SHALL serialize the Leaf_Layout dictionary to JSON with 2-space indentation.
2. THE Layout_Generator SHALL write the JSON output to a file in the output directory.
3. IF the output directory does not exist, THEN THE Layout_Generator SHALL create the directory (including parent directories).
4. WHEN a Leaf_Layout is successfully written, THE Layout_Generator SHALL print the output file path to stdout.
5. IF an error occurs while writing the output file, THEN THE Layout_Generator SHALL print a descriptive error to stderr and continue processing remaining entries.

### Requirement 10: Batch Processing

**User Story:** As a layout author, I want the tool to process a directory of chronicle PDFs in a single invocation, so that I can generate layouts for an entire season at once.

#### Acceptance Criteria

1. WHEN `pdf_path` is a directory, THE Layout_Generator SHALL recursively find all `.pdf` files in all subdirectories and process each one that has a matching entry in the Metadata_File.
2. WHEN `pdf_path` is a single file, THE Layout_Generator SHALL process only that file (looking up its entry in the Metadata_File).
3. IF a Chronicle_PDF found in the directory has no matching entry in the Metadata_File, THEN THE Layout_Generator SHALL print a warning to stderr and skip that file.
4. IF an error occurs while processing one Chronicle_PDF, THEN THE Layout_Generator SHALL print the error to stderr and continue processing remaining files.
5. WHEN all files have been processed, THE Layout_Generator SHALL print a summary indicating the number of layouts generated and the number of files skipped.
6. THE Layout_Generator SHALL exit with code 0 when at least one layout is generated successfully, and exit with code 1 when no layouts are generated.

### Requirement 11: Safe Label Generation

**User Story:** As a developer, I want item and checkbox text sanitized into preset-safe labels, so that preset names are valid JSON keys and consistent across runs.

#### Acceptance Criteria

1. THE Layout_Generator SHALL generate Safe_Labels by replacing spaces with underscores.
2. THE Layout_Generator SHALL strip the following characters from Safe_Labels: commas, periods, parentheses, single quotes, and double quotes.
3. THE Layout_Generator SHALL truncate Safe_Labels to 50 characters.
4. FOR ALL input text strings, THE Layout_Generator SHALL produce the same Safe_Label when given the same input (deterministic).

### Requirement 12: Round-Trip JSON Consistency

**User Story:** As a developer, I want the generated layout JSON to be round-trip stable, so that serializing and deserializing produces an equivalent dictionary.

#### Acceptance Criteria

1. THE Leaf_Layout produced by the Layout_Generator SHALL contain only Python built-in types: `dict`, `list`, `float`, `int`, and `str`.
2. FOR ALL valid Leaf_Layout outputs, serializing to JSON with `json.dumps` and deserializing with `json.loads` SHALL produce a dictionary equal to the original output.

### Requirement 13: Error Handling

**User Story:** As a layout author, I want clear error messages when something goes wrong, so that I can identify and fix issues in my metadata or PDF files.

#### Acceptance Criteria

1. IF the layouts directory does not exist, THEN THE Layout_Generator SHALL print a descriptive error to stderr and exit with a non-zero exit code.
2. IF a Chronicle_PDF cannot be opened by PyMuPDF, THEN THE Layout_Generator SHALL print a descriptive error to stderr identifying the file and skip that entry.
3. THE Layout_Generator SHALL print all error and warning messages to stderr.
4. THE Layout_Generator SHALL include the Metadata_File entry identifier (layout id or PDF path) in all error and warning messages for that entry.

### Requirement 14: Programmatic API

**User Story:** As a developer, I want a public function that accepts a PDF path and metadata and returns the layout dictionary, so that the generation logic is testable and reusable outside the CLI.

#### Acceptance Criteria

1. THE Layout_Generator SHALL expose a public function `generate_layout_json` that accepts a Chronicle_PDF path, extraction region coordinates, canvas names, and metadata fields, and returns the Leaf_Layout as a Python dictionary.
2. THE `generate_layout_json` function SHALL handle the complete pipeline internally: text extraction, item segmentation, checkbox detection, checkbox label extraction, and layout assembly.
3. IF the Chronicle_PDF path does not exist, THEN THE `generate_layout_json` function SHALL raise a `FileNotFoundError` with a descriptive message.

### Requirement 15: Regex Substitution in Templates

**User Story:** As a layout author, I want to use regex capture groups in my metadata templates, so that a single rule can generate correct ids, parents, and paths for many chronicles.

#### Acceptance Criteria

1. THE Layout_Generator SHALL replace `$1`, `$2`, etc. in template strings with the corresponding capture group values from the rule's `pattern` match.
2. THE Layout_Generator SHALL replace `$0` with the entire matched string.
3. IF a template references a capture group that does not exist in the match (e.g., `$3` when the pattern has only 2 groups), THEN THE Layout_Generator SHALL leave the reference as a literal string and print a warning to stderr.
4. THE Layout_Generator SHALL perform substitution on the `id`, `parent`, `description`, and `default_chronicle_location` fields of the matched rule.
